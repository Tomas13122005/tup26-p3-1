# Apunte de clase: Componentes en Blazor desde primeros principios

## Objetivo

Entender cómo se construyen componentes en Blazor partiendo de un único modelo mental:

```txt
estado + plantilla = interfaz renderizada
```

Vamos a usar **una sola aplicación como hilo conductor**: una agenda de contactos. Cada concepto nuevo agrega una pieza a esa misma app, en vez de empezar un ejemplo distinto cada vez.

Temas:

- Qué es un componente y cómo crear archivos `.razor`.
- Parámetros con `[Parameter]` (padre → hijo).
- Componentes anidados y reutilización del modelo de datos.
- `ChildContent` y `RenderFragment` (contenido interno).
- Cascading parameters (datos transversales).
- Estilos: globales, aislados y parametrizados.
- Data binding con `@bind`.
- Comunicación hijo → padre con `EventCallback<T>`.
- Two-way binding entre componentes propios.
- Métodos del ciclo de vida.
- Ejemplo integrador maestro-detalle.

---

# 1. Modelo mental de Blazor

Un componente combina markup Razor, estado, código C#, parámetros de entrada, eventos y, opcionalmente, estilos propios.

La idea central es una sola:

```txt
UI = f(estado)
```

El ciclo es:

```txt
estado + plantilla Razor
        ↓
Blazor construye la UI
        ↓
el usuario hace algo (click, input, submit)
        ↓
cambia el estado
        ↓
Blazor vuelve a renderizar solo lo necesario
```

No es magia: es una función con buena prensa. Todo lo que sigue son distintas formas de mover datos hacia ese estado y hacia la plantilla.

# 2. El modelo de datos compartido

Antes de crear componentes, definimos el dato que toda la app va a manejar. En una aplicación real **el modelo vive en un solo lugar**, no duplicado dentro de cada componente.

```csharp
// Models/Contact.cs

namespace AgendaBlazor.Models;

public record Contact(
    int Id,
    string FullName,
    string Email,
    string Phone
);
```

Cualquier componente que lo use solo necesita:

```razor
@using AgendaBlazor.Models
```

A lo largo del apunte asumimos que este `using` está disponible (por ejemplo, agregado en `_Imports.razor`).

# 3. Crear un componente mínimo

Un componente vive en un archivo `.razor` y tiene dos partes: la visual y la lógica.

```razor
<!-- Components/ContactCard.razor -->

<article class="card">
    <h3>@Contact.FullName</h3>
    <p>@Contact.Email</p>
    <p>@Contact.Phone</p>
</article>

@code {
    [Parameter]
    public Contact Contact { get; set; } = default!;
}
```

La anotación `[Parameter]` significa que esa propiedad **puede recibir un valor desde afuera**. Se usa así:

```razor
<ContactCard Contact="ana" />
```

Conceptualmente es como llamar a una función, pero en lugar de devolver un valor, produce UI:

```csharp
ContactCard(Contact: ana)   // → markup renderizado
```

> Se puede recibir el contacto entero (`Contact`) o campos sueltos (`FullName`, `Email`...). Pasar el objeto suele ser más limpio cuando van juntos.

La regla más importante de Blazor aparece ya acá:

```txt
El padre suele ser dueño del estado.
El hijo recibe datos y renderiza.
```

# 4. Componentes anidados

Un componente puede usar otros adentro (*nested components*). Nuestra agenda tendrá esta jerarquía:

```txt
Contacts.razor
 └── ContactList.razor
      └── ContactCard.razor (x N)
```

La página dueña de los datos:

```razor
<!-- Pages/Contacts.razor -->

@page "/contacts"

<h1>Agenda</h1>

<ContactList Contacts="contacts" />

@code {
    private List<Contact> contacts =
    [
        new(1, "Ana Gómez", "ana@email.com", "3815551111"),
        new(2, "Luis Díaz", "luis@email.com", "3815552222"),
        new(3, "María Ruiz", "maria@email.com", "3815553333")
    ];
}
```

La lista, que recibe los datos y crea una tarjeta por contacto:

```razor
<!-- Components/ContactList.razor -->

<ul>
    @foreach (var contact in Contacts) {
        <li @key="contact.Id">
            <ContactCard Contact="contact" />
        </li>
    }
</ul>

@code {
    [Parameter]
    public IReadOnlyList<Contact> Contacts { get; set; } = [];
}
```

`@key="contact.Id"` ayuda a Blazor a identificar cada elemento de la lista entre renders, lo que evita reordenamientos incorrectos.

El flujo de datos siempre es **padre → hijo**: el padre tiene el estado, el hijo lo muestra.

# 5. `ChildContent`: componentes que reciben contenido interno

A veces queremos un componente que envuelva contenido escrito por quien lo usa:

```razor
<Panel Title="Datos del contacto">
    <p>Nombre: Ana Gómez</p>
    <p>Teléfono: 3815551111</p>
</Panel>
```

Para lograrlo, el componente expone una propiedad `ChildContent` de tipo `RenderFragment`:

```razor
<!-- Components/Panel.razor -->

<section class="panel">
    <header><h2>@Title</h2></header>
    <main>@ChildContent</main>
</section>

@code {
    [Parameter]
    public string Title { get; set; } = "";

    [Parameter]
    public RenderFragment? ChildContent { get; set; }
}
```

```txt
RenderFragment = un pedazo de interfaz que otro componente puede renderizar
```

Esto permite componentes estructurales reutilizables:

```razor
<Modal Title="Editar contacto">
    <ContactEditor Contact="selectedContact" />
</Modal>
```

# 6. Cascading parameters: datos transversales

Imaginá que muchos componentes necesitan el mismo dato (el tema visual):

```txt
ContactsPage → ContactList → ContactCard
```

Pasarlo manualmente nivel por nivel (`Theme="@Theme"` en cada uno) produce **parameter drilling**: componentes intermedios reciben parámetros que no usan, solo para reenviarlos.

La solución: un ancestro **publica** un valor y cualquier descendiente lo **consume** sin que los intermedios lo toquen.

El valor a compartir:

```csharp
// Models/AppTheme.cs

namespace AgendaBlazor.Models;

public class AppTheme {
    public string Mode { get; set; } = "light";
    public string AccentColor { get; set; } = "#2563eb";
}
```

El ancestro publica con `<CascadingValue>`:

```razor
@page "/contacts"

<CascadingValue Value="theme">
    <h1>Agenda</h1>
    <ContactList Contacts="contacts" />
</CascadingValue>

@code {
    private AppTheme theme = new() { Mode = "dark", AccentColor = "#7c3aed" };

    private List<Contact> contacts =
    [
        new(1, "Ana Gómez", "ana@email.com", "3815551111"),
        new(2, "Luis Díaz", "luis@email.com", "3815552222")
    ];
}
```

El descendiente consume con `[CascadingParameter]` (sin que `ContactList` haga nada):

```razor
<!-- Components/ContactCard.razor -->

<article class="card" style="border-color:@Theme?.AccentColor">
    <h3>@Contact.FullName</h3>
    <p>@Contact.Email</p>
    <small>Tema: @Theme?.Mode</small>
</article>

@code {
    [Parameter]
    public Contact Contact { get; set; } = default!;

    [CascadingParameter]
    public AppTheme? Theme { get; set; }
}
```

Resumen:

```txt
[Parameter]           → dato explícito del padre al hijo directo
[CascadingParameter]  → dato compartido de un ancestro a sus descendientes
```

Conviene para datos transversales: tema visual, usuario actual, idioma/cultura, estado global de layout. **No** para datos comunes: si todo es cascading, la app termina pareciendo una búsqueda del tesoro con bugs incluidos.

# 7. Estilos de componentes

## 7.1 CSS global

En `wwwroot/app.css`:

```css
.card {
    border: 1px solid #ddd;
    border-radius: 12px;
    padding: 1rem;
}
```

Afecta a toda la app. Sirve para `body`, tipografía, layout principal, botones base, variables CSS y utilitarios comunes.

## 7.2 CSS aislado por componente

Blazor soporta **CSS isolation**: un archivo `.razor.css` con el mismo nombre que el componente.

```txt
Components/
 ├── ContactCard.razor
 └── ContactCard.razor.css
```

```css
/* Components/ContactCard.razor.css */
.card {
    border: 1px solid #ddd;
    border-radius: 12px;
    padding: 1rem;
    background: white;
}

.card h3 { margin: 0 0 .5rem; }
.card p  { margin: .25rem 0; }
```

Los estilos quedan asociados al componente y se reducen las colisiones con el resto de la app.

## 7.3 Estilos parametrizados

A veces el padre necesita pasar una clase o estilo:

```razor
<article class="card @Class" style="@Style">
    <h3>@Contact.FullName</h3>
</article>

@code {
    [Parameter] public Contact Contact { get; set; } = default!;
    [Parameter] public string? Class { get; set; }
    [Parameter] public string? Style { get; set; }
}
```

Uso:

```razor
<ContactCard Contact="ana" Class="selected" />
```

No abuses de `Style` inline: si cada componente recibe CSS suelto, la app empieza a parecer una mudanza sin cajas etiquetadas.

# 8. Data binding

**Data binding** es vincular un dato C# con un elemento visual. Se usa la directiva `@bind`:

```razor
<input @bind="name" />
<p>Hola, @name</p>

@code {
    private string name = "";
}
```

Cuando cambia el input cambia la variable; cuando cambia la variable cambia la UI.

**Actualizar en cada tecla.** Por defecto muchos inputs actualizan con `onchange` (al perder el foco). Para reaccionar en cada tecla:

```razor
<input @bind="name" @bind:event="oninput" />
```

**Binding con objetos.** Funciona igual sobre propiedades de un objeto:

```razor
<input @bind="contact.FullName" @bind:event="oninput" />
<input @bind="contact.Email"    @bind:event="oninput" />

<ContactCard Contact="contact" />   <!-- vista previa en vivo -->

@code {
    private Contact contact = new(1, "Ana Gómez", "ana@email.com", "3815551111");
}
```

El flujo: el input cambia → cambia `contact.FullName` → Blazor renderiza → `ContactCard` recibe el nuevo valor.

**`@bind:get` y `@bind:set`.** Para controlar lectura y escritura por separado (por ejemplo, limitar a 20 caracteres):

```razor
<input @bind:get="name" @bind:set="SetName" @bind:event="oninput" />

@code {
    private string name = "";

    private void SetName(string value) =>
        name = value.Length <= 20 ? value : value[..20];
}
```

```txt
@bind:get → de dónde sale el valor actual
@bind:set → qué hago cuando el usuario lo cambia
```

# 9. Comunicación entre componentes

Las cuatro vías, que ya vimos por separado:

```txt
Padre → Hijo:            [Parameter]
Hijo → Padre:            EventCallback<T>
Ancestro → Descendientes: [CascadingParameter]
Input ↔ Estado:          @bind
```

Falta la que va de **hijo a padre**.

## 9.1 Hijo → padre con `EventCallback<T>`

Cuando el hijo quiere avisarle algo al padre (por ejemplo, "me seleccionaron"), recibe un callback como parámetro y lo invoca:

```razor
<!-- Components/ContactListItem.razor -->

<li>
    <button @onclick="Select">@Contact.FullName</button>
</li>

@code {
    [Parameter]
    public Contact Contact { get; set; } = default!;

    [Parameter]
    public EventCallback<Contact> OnSelected { get; set; }

    private async Task Select() => await OnSelected.InvokeAsync(Contact);
}
```

El padre pasa el dato y el callback, y reacciona:

```razor
@page "/communication-demo"

<ul>
    @foreach (var contact in contacts) {
        <ContactListItem Contact="contact" OnSelected="HandleSelected" />
    }
</ul>

@if (selectedContact is not null) {
    <ContactCard Contact="selectedContact" />
}

@code {
    private Contact? selectedContact;

    private List<Contact> contacts =
    [
        new(1, "Ana Gómez", "ana@email.com", "3815551111"),
        new(2, "Luis Díaz", "luis@email.com", "3815552222")
    ];

    private void HandleSelected(Contact contact) => selectedContact = contact;
}
```

Flujo:

```txt
Padre pasa datos + callback → usuario hace click en el hijo
→ el hijo invoca el callback → el padre actualiza su estado → Blazor re-renderiza
```

Este patrón es central: **el hijo avisa, el padre decide**.

## 9.2 Two-way binding entre componentes propios

`@bind` también funciona con componentes propios. La convención es exponer una propiedad `X` y un `EventCallback<T>` llamado `XChanged`:

```razor
<!-- Components/SearchBox.razor -->

<input value="@Value" @oninput="HandleInput" placeholder="@Placeholder" />

@code {
    [Parameter]
    public string Value { get; set; } = "";

    [Parameter]
    public EventCallback<string> ValueChanged { get; set; }

    [Parameter]
    public string Placeholder { get; set; } = "Buscar...";

    private async Task HandleInput(ChangeEventArgs e) =>
        await ValueChanged.InvokeAsync(e.Value?.ToString() ?? "");
}
```

Gracias al par `Value` / `ValueChanged`, se usa así:

```razor
<SearchBox @bind-Value="searchText" Placeholder="Buscar contacto..." />
```

Que es equivalente a escribir a mano:

```razor
<SearchBox Value="@searchText"
           ValueChanged="value => searchText = value" />
```

# 10. Métodos del ciclo de vida

Un componente nace, recibe parámetros, se renderiza y muere. Blazor permite engancharse en cada momento. Los métodos principales:

```csharp
OnInitialized() / OnInitializedAsync()      // nace
OnParametersSet() / OnParametersSetAsync()  // recibe o cambian parámetros
ShouldRender()                              // ¿debe re-renderizar?
OnAfterRender(bool firstRender) / ...Async  // ya se renderizó
Dispose() / DisposeAsync()                  // se limpia al morir
```

**`OnInitialized` / `OnInitializedAsync`** — inicializar estado al nacer. La versión async sirve para cargar datos:

```razor
@if (contacts is null) {
    <p>Cargando...</p>
} else {
    <ul>@foreach (var c in contacts) { <li>@c.FullName</li> }</ul>
}

@code {
    private List<Contact>? contacts;

    protected override async Task OnInitializedAsync() {
        // En una app real: contacts = await ContactService.GetAllAsync();
        await Task.Delay(500);
        contacts = [ new(1, "Ana Gómez", "ana@email.com", "3815551111") ];
    }
}
```

**`OnParametersSet`** — recalcular estado derivado cuando llegan o cambian parámetros:

```razor
<h3>@DisplayName</h3>

@code {
    [Parameter] public string FirstName { get; set; } = "";
    [Parameter] public string LastName { get; set; } = "";

    private string DisplayName { get; set; } = "";

    protected override void OnParametersSet() =>
        DisplayName = $"{LastName}, {FirstName}";
}
```

**`ShouldRender`** — decide si re-renderizar. Casi nunca hace falta; es optimización fina. Si lo usás sin medir, estás afinando un violín con una llave inglesa.

**`OnAfterRender(bool firstRender)`** — corre después del render, cuando el DOM ya existe. Sirve para interactuar con JavaScript, medir elementos, o inicializar librerías externas:

```razor
@code {
    protected override void OnAfterRender(bool firstRender) {
        if (firstRender)
            Console.WriteLine("Primer render completo");
    }
}
```

```txt
firstRender == true  → primera renderización
firstRender == false → renderizaciones posteriores
```

**`Dispose`** — limpiar recursos. Si creás timers, suscripciones o recursos externos, tenés que liberarlos:

```razor
@implements IDisposable

<p>Hora: @currentTime</p>

@code {
    private Timer? timer;
    private string currentTime = "";

    protected override void OnInitialized() {
        timer = new Timer(_ => {
            currentTime = DateTime.Now.ToLongTimeString();
            InvokeAsync(StateHasChanged);
        }, null, 0, 1000);
    }

    public void Dispose() => timer?.Dispose();
}
```

Si no limpiás, el componente desaparece de la pantalla pero deja procesos vivos: fantasmas con consumo de RAM.

# 11. Ejemplo integrador: agenda maestro-detalle

Ahora unimos todo en una mini agenda con búsqueda, lista seleccionable y panel de detalle.

```txt
Components/
 ├── SearchBox.razor        (ya lo tenemos: §9.2)
 ├── ContactList.razor      (selección con EventCallback)
 └── ContactDetails.razor   (panel de detalle)
Pages/
 └── Contacts.razor         (dueña del estado)
```

## 11.1 `ContactList` con selección

```razor
<!-- Components/ContactList.razor -->

<ul class="contact-list">
    @foreach (var contact in Contacts) {
        <li @key="contact.Id">
            <button
                class="contact-list-item @(contact.Id == SelectedContactId ? "selected" : "")"
                @onclick="() => OnSelected.InvokeAsync(contact)">
                @contact.FullName
            </button>
        </li>
    }
</ul>

@code {
    [Parameter] public IReadOnlyList<Contact> Contacts { get; set; } = [];
    [Parameter] public int? SelectedContactId { get; set; }
    [Parameter] public EventCallback<Contact> OnSelected { get; set; }
}
```

## 11.2 `ContactDetails` (puramente visual)

```razor
<!-- Components/ContactDetails.razor -->

@if (Contact is null) {
    <section class="empty-details"><p>Seleccioná un contacto.</p></section>
} else {
    <section class="contact-details">
        <h2>@Contact.FullName</h2>
        <p><strong>Email:</strong> @Contact.Email</p>
        <p><strong>Teléfono:</strong> @Contact.Phone</p>
    </section>
}

@code {
    [Parameter] public Contact? Contact { get; set; }
}
```

## 11.3 Página `Contacts` (orquesta todo)

```razor
<!-- Pages/Contacts.razor -->

@page "/contacts"

<h1>Agenda</h1>

<div class="layout">
    <aside>
        <SearchBox @bind-Value="searchText" Placeholder="Buscar contacto..." />
        <ContactList
            Contacts="filteredContacts"
            SelectedContactId="selectedContact?.Id"
            OnSelected="SelectContact" />
    </aside>

    <main>
        <ContactDetails Contact="selectedContact" />
    </main>
</div>

@code {
    private string searchText = "";
    private Contact? selectedContact;

    private List<Contact> contacts =
    [
        new(1, "Ana Gómez", "ana@email.com", "3815551111"),
        new(2, "Luis Díaz", "luis@email.com", "3815552222"),
        new(3, "María Ruiz", "maria@email.com", "3815553333"),
        new(4, "Carlos Medina", "carlos@email.com", "3815554444")
    ];

    private IReadOnlyList<Contact> filteredContacts =>
        contacts
            .Where(c =>
                c.FullName.Contains(searchText, StringComparison.OrdinalIgnoreCase) ||
                c.Email.Contains(searchText, StringComparison.OrdinalIgnoreCase) ||
                c.Phone.Contains(searchText, StringComparison.OrdinalIgnoreCase))
            .ToList();

    protected override void OnInitialized() =>
        selectedContact = contacts.FirstOrDefault();

    private void SelectContact(Contact contact) => selectedContact = contact;
}
```

Cada concepto del apunte aparece acá:

```txt
Anidación:        Contacts → SearchBox / ContactList / ContactDetails
Padre → hijo:     Contacts pasa Contacts, SelectedContactId, Contact
Hijo → padre:     ContactList avisa con OnSelected
Two-way binding:  SearchBox con @bind-Value
Lifecycle:        OnInitialized elige el contacto inicial
Estado:           searchText y selectedContact
Estado derivado:  filteredContacts (se recalcula al escribir)
```

# 12. Cómo diseñar un componente

Antes de escribirlo, respondé:

```txt
1. ¿Qué muestra?
2. ¿Qué datos necesita?
3. ¿Quién es dueño del estado?
4. ¿Tiene estado interno o todo viene de afuera?
5. ¿Debe avisar eventos al padre?
6. ¿Debe recibir contenido interno?
7. ¿Sus estilos son propios o globales?
```

Aplicado a `ContactCard`:

```txt
- Muestra un contacto.
- Recibe Contact desde afuera.
- No carga datos por su cuenta ni decide la selección.
- Puede avisar OnSelected.
- Puede tener CSS aislado.
```

# 13. Regla de oro

```txt
El padre posee el estado.
El hijo muestra datos.
El hijo avisa eventos.
El padre decide qué hacer.
```

Las cuatro vías de comunicación:

```txt
Padre → Hijo:             [Parameter]
Hijo → Padre:             EventCallback<T>
Ancestro → Descendientes: [CascadingParameter]
Input ↔ Estado:           @bind
```

El corazón conceptual es: **estado, parámetros, eventos, renderizado y composición**. Lo que sigue (todo construido sobre esto) sería: formularios con `EditForm` y validaciones, servicios inyectados, render modes, componentes genéricos, templates con `RenderFragment<T>`, optimización de renderizado y JS interop.
