## 1. ¿Por qué existe `EditForm`?

Antes de aprender la sintaxis, pensemos el problema que resuelve.

Un formulario web "crudo" en HTML te da un `<form>` con inputs, pero vos tenés que resolver a mano todo lo demás:

- Leer cada valor del input y meterlo en tu objeto.
- Validar que el email tenga formato de email, que la edad sea un número, que el nombre no esté vacío.
- Mostrar los mensajes de error al lado del campo correcto.
- Evitar que el formulario se mande si hay errores.
- Volver a poblar los inputs cuando editás un registro existente.

Eso es mucho código repetitivo y propenso a errores. `EditForm` es el componente de Blazor que te da todo eso **enganchado a un objeto C#**: vos le pasás un modelo, decorás las propiedades con atributos de validación, y Blazor se encarga del binding, la validación y el flujo de envío.

La idea central: **el formulario está atado a un objeto del dominio, no a strings sueltos**. Vos pensás en términos de `Contacto`, no de "el valor del input nombre".

---

## 2. Conceptos centrales

`EditForm` se apoya en tres piezas que conviene tener claras desde el arranque.

### 2.1. El modelo (`Model`)

Es el objeto C# que el formulario edita. Cada propiedad pública se puede atar a un input. La validación vive acá, como atributos sobre las propiedades (Data Annotations):

```csharp
using System.ComponentModel.DataAnnotations;

public class Contacto {
    [Required(ErrorMessage = "El nombre es obligatorio")]
    [StringLength(50, ErrorMessage = "Máximo 50 caracteres")]
    public string Nombre { get; set; } = "";

    [Required(ErrorMessage = "El email es obligatorio")]
    [EmailAddress(ErrorMessage = "Email inválido")]
    public string Email { get; set; } = "";

    [Range(0, 120, ErrorMessage = "Edad entre 0 y 120")]
    public int Edad { get; set; }
}
```

### 2.2. El `EditContext`

Acá está la parte que casi nadie explica y que conviene entender, porque es lo que hace que el formulario "reaccione".

Cuando le pasás un `Model` a `EditForm`, internamente Blazor crea un `EditContext`. El `EditContext` es un objeto que **envuelve al modelo y lleva la cuenta del estado del formulario**: qué campos fueron tocados, qué campos son válidos, si hubo algún cambio. Es el cerebro que coordina todo.

Tenés dos formas de trabajar:

- Pasarle `Model` y dejar que Blazor cree el `EditContext` solo (lo más común).
- Crear vos el `EditContext` y pasárselo con el parámetro `EditContext` (cuando necesitás control fino: validar a mano, escuchar cambios de campo, etc.).

**No pases los dos a la vez.** O `Model`, o `EditContext`, nunca ambos.

### 2.3. Los componentes de input

Blazor trae inputs propios que se enganchan al `EditContext`: `InputText`, `InputNumber`, `InputCheckbox`, `InputSelect`, `InputDate`, `InputTextArea`, etc. Cada uno se ata a una propiedad del modelo con `@bind-Value`:

```razor
<InputText @bind-Value="contacto.Nombre" />
```

Esto es binding bidireccional: si el usuario escribe, la propiedad se actualiza; si la propiedad cambia desde código, el input se actualiza. La diferencia con un `<input>` común es que estos avisan al `EditContext` cuándo cambiaron, y por eso la validación funciona.

## 3. Dinámica de funcionamiento

Veamos el flujo completo, paso a paso, desde que se renderiza hasta que se envía.

1. **Render inicial.** `EditForm` recibe el `Model`, crea el `EditContext` y renderiza los inputs. Cada `InputText`/`InputNumber` se registra contra el `EditContext` y muestra el valor actual de su propiedad.

2. **El usuario escribe.** Al cambiar un input, `@bind-Value` actualiza la propiedad del modelo y el input le avisa al `EditContext` que ese campo se modificó. El `EditContext` lo marca como "tocado".

3. **Validación.** Esto lo dispara el `<DataAnnotationsValidator />` (un componente que tenés que poner dentro del form). Él escucha al `EditContext` y, cuando un campo cambia o cuando se intenta enviar, corre los atributos de validación y guarda los resultados.

4. **Feedback visual.** `<ValidationMessage For="..." />` muestra el mensaje de error de un campo puntual; `<ValidationSummary />` muestra todos juntos. Además, los inputs reciben clases CSS (`valid` / `invalid`, `modified`) que podés estilar.

5. **Envío.** Cuando el usuario aprieta el botón submit, `EditForm` revalida todo y dispara **uno** de tres callbacks:
   - `OnValidSubmit` — el modelo pasó todas las validaciones.
   - `OnInvalidSubmit` — hubo errores.
   - `OnSubmit` — se dispara siempre, y vos validás a mano (no uses este junto con los otros dos).

En la práctica, el 90% de las veces usás `OnValidSubmit` y listo: si llega ahí, el modelo es válido.

## 4. Ejemplo completo

Formulario de alta de contacto. Notá cómo el componente solo se preocupa por el modelo y el callback; toda la maquinaria de validación está declarada.

```razor
@page "/contacto-nuevo"

@using System.ComponentModel.DataAnnotations

<h3>Nuevo contacto</h3>

<EditForm Model="this.contacto" OnValidSubmit="this.Guardar">
    <DataAnnotationsValidator />
    <ValidationSummary />

    <div class="mb-3">
        <label class="form-label">Nombre</label>
        <InputText class="form-control" @bind-Value="this.contacto.Nombre" />
        <ValidationMessage For="@(() => this.contacto.Nombre)" />
    </div>

    <div class="mb-3">
        <label class="form-label">Email</label>
        <InputText class="form-control" @bind-Value="this.contacto.Email" />
        <ValidationMessage For="@(() => this.contacto.Email)" />
    </div>

    <div class="mb-3">
        <label class="form-label">Edad</label>
        <InputNumber class="form-control" @bind-Value="this.contacto.Edad" />
        <ValidationMessage For="@(() => this.contacto.Edad)" />
    </div>

    <button type="submit" class="btn btn-primary">Guardar</button>
</EditForm>

@if (this.guardado) {
    <div class="alert alert-success mt-3">
        Contacto guardado: @this.contacto.Nombre (@this.contacto.Email)
    </div>
}

@code {
    private Contacto contacto = new();
    private bool guardado = false;

    private void Guardar() {
        // Si llegamos acá, el modelo ya pasó todas las validaciones.
        this.guardado = true;
        // Acá iría la llamada al repositorio / DbContext.
    }

    public class Contacto {
        [Required(ErrorMessage = "El nombre es obligatorio")]
        [StringLength(50, ErrorMessage = "Máximo 50 caracteres")]
        public string Nombre { get; set; } = "";

        [Required(ErrorMessage = "El email es obligatorio")]
        [EmailAddress(ErrorMessage = "Email inválido")]
        public string Email { get; set; } = "";

        [Range(18, 120, ErrorMessage = "La edad debe estar entre 18 y 120")]
        public int Edad { get; set; }
    }
}
```

### Qué mirar de este ejemplo

- El `Model` apunta a `this.contacto`, una instancia viva. Blazor escribe directo sobre ese objeto.
- `DataAnnotationsValidator` es lo que conecta los atributos `[Required]`, `[EmailAddress]`, etc. con el `EditContext`. **Sin él, no se valida nada.** Es el olvido más común.
- `ValidationMessage For="@(() => this.contacto.Nombre)"` usa una expresión lambda para identificar el campo. Esa sintaxis rara con `() =>` no es decorativa: Blazor analiza la expresión para saber a qué propiedad apunta.
- `OnValidSubmit` solo corre si todo validó. Por eso adentro de `Guardar()` no hace falta volver a chequear nada.

## 5. Validación manual con `EditContext` (cuando lo necesitás)

A veces querés validar fuera del submit (por ejemplo, habilitar el botón solo si el form es válido). Ahí creás vos el `EditContext`:

```razor
<EditForm EditContext="this.editContext" OnValidSubmit="this.Guardar">
    <DataAnnotationsValidator />
    <!-- inputs... -->
    <button type="submit" class="btn btn-primary" disabled="@(!this.esValido)">
        Guardar
    </button>
</EditForm>

@code {
    private Contacto contacto = new();
    private EditContext editContext = default!;
    private bool esValido = false;

    protected override void OnInitialized() {
        this.editContext = new EditContext(this.contacto);
        this.editContext.OnFieldChanged += (sender, args) => {
            this.esValido = this.editContext.Validate();
        };
    }

    private void Guardar() { /* ... */ }
}
```

Acá enganchás el evento `OnFieldChanged` del `EditContext` para revalidar cada vez que un campo cambia, y usás el resultado para habilitar o deshabilitar el botón. Esto es exactamente lo que `EditContext` te permite y un `<form>` crudo no.

---

## 6. Errores comunes

- **Olvidar `<DataAnnotationsValidator />`.** El formulario "funciona" pero nunca valida. Siempre llega a `OnValidSubmit`.
- **Pasar `Model` y `EditContext` juntos.** Blazor tira excepción. Elegí uno.
- **Usar `<input>` de HTML en vez de `InputText`.** Los inputs comunes no avisan al `EditContext`, así que la validación los ignora.
- **Mezclar `OnSubmit` con `OnValidSubmit`/`OnInvalidSubmit`.** O usás el manual, o usás los automáticos.
- **Modelo con propiedades que no son `{ get; set; }` públicas.** El binding necesita acceso de lectura y escritura.

---

## 7. Resumen mental

Pensalo así: `EditForm` es un `<form>` que sabe de C#. Le das un objeto, declarás las reglas como atributos, ponés el `DataAnnotationsValidator` para que esas reglas se apliquen, y dejás que el `EditContext` orqueste el estado. Tu código solo se ocupa de qué hacer cuando el modelo ya es válido.
